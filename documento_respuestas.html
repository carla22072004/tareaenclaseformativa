Flags de seguridad de cookies de sesion
Los tres flags que debe tener siempre la cookie de sesion en produccion:
  • HttpOnly: impide que JavaScript del cliente acceda a la cookie; previene robo de sesion via XSS.
  • Secure: la cookie solo se envia por HTTPS; previene interceptacion en redes inseguras.
  • SameSite=Strict: la cookie no se envia en solicitudes originadas desde otros sitios; previene CSRF.

Por que el PFC usa JWT en lugar de sesiones de servidor

Las sesiones de servidor funcionan bien para aplicaciones web tradicionales (multi-page). El PFC usa una arquitectura diferente: el frontend es una SPA Angular que consume una API REST de Spring Boot. Esta arquitectura tiene tres caracteristicas que hacen que las sesiones de servidor sean inadecuadas y que JWT sea la solucion correcta.

Problema Sesion de servidor (no usada en el PFC) JWT en cookie HttpOnly (PFC)
Estado en el servidor El servidor debe almacenar la sesion; complica el escalado horizontal (multiples instancias) El servidor es stateless: no almacena nada; cualquier instancia puede validar el token
API REST y SPA Las APIs REST no deben usar cookies de sesion de forma nativa; el frontend necesita manejar CORS y credenciales manualmente El JWT viaja en el encabezado Authorization: Bearer o en cookie HttpOnly; compatible con CORS en Angular
Revocacion Sencilla: eliminar la sesion del servidor Requiere blacklist en Redis (como se implementa en el PFC con el JTI)
Seguridad del token Cookie de sesion: si se roba el ID, el atacante tiene acceso hasta que expire JWT en cookie HttpOnly: JavaScript no puede leerla; accessToken de corta vida (1h); refreshToken en cookie separada
Conclusion practica
Para una aplicacion web tradicional multi-pagina (como la que se construye en esta practica), las sesiones de servidor son la herramienta correcta. Para una API REST consumida por una SPA Angular (como el PFC), JWT almacenado en cookie HttpOnly es la solucion correcta porque mantiene la naturaleza stateless de REST, escala horizontalmente sin configuracion adicional, y protege el token contra ataques XSS (JavaScript no puede leer una cookie HttpOnly).
